# НАУЧНО-ТЕХНИЧЕСКИЙ ОТЧЕТ О НИР

## Тема
Прототип системы агентно-ориентированного моделирования взаимодействия объекта с атмосферной средой в 3D.

## Реферат

Отчет описывает разработку интерактивного программного прототипа, моделирующего движение дискретных частиц воздуха вокруг 3D-объекта с учетом базовых физических эффектов и пользовательского управления параметрами в реальном времени.

**Цель работы**: создать модульную систему визуализации и расчета, включающую:
- загрузку 3D-моделей из стандартных форматов;
- инициализацию частиц-агентов с параметрами состояния;
- расчет взаимодействий частица-частица и частица-объект;
- интерактивное управление сценой и параметрами среды;
- производительный цикл симуляции для запуска на обычном ноутбуке.

**Результаты**:
- реализована 3D-сцена на Python (ModernGL + GLFW + NumPy + Trimesh);
- частицы визуализируются инстансным рендером как 3D-сферы;
- добавлены силы среды (wind/drag/turbulence/buoyancy/gravity);
- реализованы столкновения с границами, объектом и между частицами;
- применен spatial hashing для ускорения коллизий;
- реализован интерактивный UI (ImGui + fallback Tk/Web);
- добавлены визуальные направляющие (оси, grid, контур AABB), повышающие читаемость сцены;
- выполнен headless self-check устойчивости расчетов.

## 1. Термины и определения

- **Агент-частица**: дискретный элемент среды, имеющий состояние (позиция, скорость, масса, радиус).
- **Объект-препятствие**: загруженная 3D-модель, влияющая на движение частиц.
- **AABB**: осесимметричный ограничивающий параллелепипед (axis-aligned bounding box).
- **Spatial hashing**: разбиение пространства на ячейки для ускоренного поиска потенциальных пар столкновения.
- **Simulation loop**: цикл обновления состояния системы на каждом шаге времени.
- **Headless self-check**: автоматическая проверка физического шага без открытия графического окна.

---

## 2. Введение

Модель предназначена для демонстрации принципов агентно-ориентированного моделирования атмосферной среды в инженерной постановке. В отличие от полноценных CFD-решателей, прототип ориентирован на интерактивность, наглядность и высокую скорость итераций при изменении параметров пользователем.

Практическая ценность прототипа:
- быстрый анализ качественного поведения потока;
- визуальная демонстрация взаимодействия частиц с препятствиями;
- площадка для дальнейшего усложнения физики и перехода к более строгим численным схемам.

## 3. Цель и задачи НИР

### 3.1 Цель
Разработать рабочий интерактивный прототип системы моделирования в 3D, где частицы среды взаимодействуют с объектом и между собой, а пользователь может изменять параметры процесса в реальном времени.

### 3.2 Задачи
1. Реализовать загрузку 3D-моделей из стандартных форматов.  
2. Описать состояние частиц и их инициализацию (случайно/из файла).  
3. Реализовать физическую модель и столкновения.  
4. Реализовать рендеринг и интерактивный ввод.  
5. Обеспечить приемлемую производительность на ноутбуке.  
6. Подготовить техническую документацию уровня НТО.

---

## 4. Исходные данные и ограничения

### 4.1 Технологический стек
- Python 3.10+  
- `moderngl`, `glfw`, `numpy`, `pyrr`, `trimesh`, `imgui-bundle`  

### 4.2 Физические допущения
- среда представлена дискретными частицами, а не непрерывным полем давления/скорости;
- объект для столкновений аппроксимируется AABB;
- интегрирование выполняется явной схемой по времени;
- турбулентность задается аналитическим гладким полем.

### 4.3 Ограничения текущей версии
- нет полного CFD-решения уравнений Навье-Стокса;
- коллизии с объектом упрощены (AABB вместо точного меша/BVH/SDF);
- отсутствуют граничные условия уровня инженерных аэродинамических расчетов.

## 5. Методика моделирования

1. Загружается модель объекта (или используется встроенная модель по умолчанию).  
2. Определяется расчетная область симуляции, расширенная относительно размеров объекта.  
3. Инициализируются частицы:
   - случайно в области, либо
   - по заданному JSON-конфигу (`position`, `velocity`, `mass`, `radius`).  
4. На каждом шаге времени:
   - вычисляются силы среды;
   - обрабатываются столкновения частица-частица;
   - обновляются координаты;
   - обрабатываются столкновения с границами и объектом;
   - обновляются GPU-буферы инстансного рендера.  
5. Отрисовываются:
   - grid, оси и контур AABB;
   - объект;
   - частицы;
   - UI-панель управления параметрами.

## 6. Математическая модель

### 6.1 Вектор состояния частицы
Для каждой частицы \( i \):
- позиция \( \mathbf{x}_i = (x_i, y_i, z_i) \)
- скорость \( \mathbf{v}_i = (v_{x,i}, v_{y,i}, v_{z,i}) \)
- масса \( m_i \)
- радиус \( r_i \)

### 6.2 Силы среды
Используется композиция:
- релаксация к локальному полю потока (drag);
- базовый ветер (wind vector);
- турбулентная добавка (псевдо-вихревое поле);
- плавучесть (buoyancy);
- эффективная гравитация.

Обновление скорости:
\[
\mathbf{v}_{t+\Delta t} = \mathbf{v}_t + \mathbf{a}_{flow}\Delta t
\]

Обновление позиции:
\[
\mathbf{x}_{t+\Delta t} = \mathbf{x}_t + \mathbf{v}_{t+\Delta t}\Delta t
\]

### 6.3 Столкновения
- **Частица-частица**: сферическая проверка пересечения, импульсная коррекция скорости, раздвижение с учетом масс.
- **Частица-границы**: отражение с коэффициентом восстановления.
- **Частица-объект**: столкновение с AABB объекта, отражение по нормали ближайшей грани и демпфирование касательной составляющей.

### 6.4 Устойчивость шага
Используется ограничение максимального шага `max_dt`, доступное в UI/CLI, для снижения риска численной неустойчивости при резких порывах.

## 7. Архитектура программного комплекса

### 7.1 Структура модулей
- `main.py`  
  Запуск приложения, создание окна, обработка ввода, UI, основной цикл.

- `engine/loader.py`  
  Загрузка моделей (Trimesh + fallback через Assimp), вычисление AABB, подготовка VAO.

- `engine/particles.py`  
  Хранение и обновление состояния частиц, применение управляющих воздействий, инстансный рендер 3D-сфер.

- `engine/physics.py`  
  Расчет сил среды, collisions, spatial hashing, self-check.

- `shaders/`  
  - `model.*` для объекта,
  - `particle.*` для частиц,
  - `line.*` для grid/axes/AABB-guides.

### 7.2 Принципы модульности
- разделение расчета, рендера и ввода;
- изоляция физики от GUI;
- возможность независимо развивать загрузчик модели, физику и визуализацию.

## 8. Описание реализованных алгоритмов

### 8.1 Spatial hashing
Пространство делится на 3D-ячейки. Для частицы рассматриваются только соседние ячейки (27 блоков), что уменьшает количество проверяемых пар по сравнению с полным \( O(N^2) \) перебором.

### 8.2 Инстансный рендер частиц
Геометрия одной UV-сферы используется повторно, а для каждой частицы в отдельном буфере передаются:
- центр,
- радиус,
- скорость (для цветовой кодировки).

### 8.3 Ray picking
По клику ЛКМ формируется луч из позиции курсора в мировое пространство, ищется пересечение с AABB объекта и, при попадании, применяется локальный импульс к ближайшим частицам.

## 9. Визуализация и интерфейс взаимодействия

### 9.1 Элементы визуального контекста
- **Grid**: ориентир масштаба и положения.
- **AABB-контур** (желтый): визуально показывает объем препятствия.
- **Оси координат**: X/Y/Z различаются цветами.

### 9.2 Управление
- Orbit-камера (мышь/тачпад/клавиатура),
- пауза/сброс/порыв,
- клик по объекту с локальной реакцией среды.

### 9.3 Параметрическое управление
Изменение параметров выполняется в реальном времени через встроенную ImGui-панель; при недоступности ImGui включается внешний fallback UI.

## 10. Загрузка моделей и форматы

Поддержка:
- напрямую через Trimesh: `.obj`, `.glb`, `.gltf`, и др.;
- через fallback-конвертацию Assimp: `.dae`, `.3ds` и совместимые форматы.

Добавлены тестовые модели:
- `assets/model_cube.obj`

- `assets/model_sphere.obj`

Это обеспечивает немедленный запуск и демонстрацию без поиска внешних ассетов.

## 12. Верификация и тестирование

### 12.1 Headless self-check
Команда:
```bash
python main.py --verify --particles 600 --verify-steps 480
```

Проверяется:
- отсутствие `NaN/Inf` в состоянии системы;
- корректность обработки столкновений;
- отсутствие проникновения частиц в препятствие на финальном шаге.

### 12.2 Практический smoke-test визуализации
- запуск с каждой из встроенных моделей;
- проверка реакции на клики, порывы, паузу и сброс;
- проверка изменения параметров среды через UI.

## 14. Варианты реализации (сравнение)

### Вариант А: на готовом 3D-движке/библиотеках (реализовано)
Стек: Python + ModernGL + GLFW + Trimesh.

**Плюсы**:
- быстрый цикл разработки;
- низкий порог модификации;
- доступность кода и прозрачность архитектуры.

**Минусы**:
- упрощенная физика относительно CFD;
- ограниченная точность коллизий по сложному мешу.

### Вариант Б: минимальный собственный движок
Собственный рендер-слой, собственная ECS/solver-подсистема, точная геометрическая коллизия.

**Плюсы**:
- полный контроль над математической постановкой;
- гибкость для исследовательских задач.

**Минусы**:
- высокая стоимость разработки и поддержки;
- более длительный путь до рабочего прототипа.
